I think you've found one of the strongest counterarguments to the traditional "you must understand all the code" position.

Let's take the regex example.

Suppose I give you this:

```regex
^(?=.{8,64}$)(?=.*[A-Z])(?=.*[a-z])(?=.*\d)(?=.*[^\w\s]).*$
```

Many experienced engineers cannot explain every character from memory.

Should that prevent it from shipping?

Probably not.

If you have:

* A clear requirement
* Test cases
* Edge case coverage
* Documentation of intent

Then requiring everyone to mentally parse every regex token is arguably not providing much value.

---

# What Are You Actually Approving?

This is the key distinction.

You are rarely approving:

> "I certify that I understand every implementation detail."

You are usually approving:

> "I have sufficient evidence that this behaves correctly within the system."

Those are very different statements.

For example, when reviewing a PR, I might not understand:

* A regex
* A SIMD optimization
* A complex SQL query
* A cryptographic primitive
* A lock-free data structure

Yet I may still be comfortable approving it because:

* The requirements are clear
* The implementation is isolated
* The tests are extensive
* The risk is acceptable

---

# Evidence-Based Confidence

Think about engineering confidence as coming from multiple sources:

| Source                 | Confidence |
| ---------------------- | ---------- |
| Personal understanding | High       |
| Unit tests             | Medium     |
| Property tests         | High       |
| Fuzz tests             | High       |
| Formal verification    | Very High  |
| Production telemetry   | High       |
| Industry adoption      | High       |
| Code review            | Medium     |

Traditionally, we over-indexed on personal understanding because it was often the only tool available.

Today we have much better tools.

---

# The Limits of Understanding

There's another uncomfortable truth:

Many engineers think they understand code that they actually don't.

I've seen developers confidently approve:

* concurrency code
* memory management code
* distributed systems code

because it "looked right."

Then six months later a race condition appears.

In contrast, a heavily-tested implementation they don't fully understand may actually be safer.

---

# Where Understanding Still Matters

There are areas where requirements and tests alone are often insufficient.

### Security

Example:

```rust
authenticate_user(...)
```

All tests pass.

But the implementation has a timing attack.

Requirements didn't mention timing attacks.

Tests didn't catch it.

Domain knowledge matters.

---

### Performance

Example:

```rust
for item in items {
    expensive_database_query(item);
}
```

All tests pass.

Requirements pass.

Production melts.

Someone needed to understand the implications.

---

### Architecture

Example:

An AI generates a system that works perfectly today.

But every feature addition now requires modifying 50 files.

Requirements pass.

Tests pass.

Yet maintainability is poor.

# The Regex Case Specifically

For a complicated regex, I would generally say:

**No, an engineer does not need to understand every symbol.**

They should understand:

* What problem it solves
* What inputs it accepts
* What inputs it rejects
* How performance behaves
* Whether tests demonstrate correctness

For example:

```text
Requirement:
Password must be:
- 8-64 characters
- At least one uppercase
- At least one lowercase
- At least one number
- At least one special character
```

Plus:

```text
✓ Test valid examples
✓ Test invalid examples
✓ Test boundaries
✓ Fuzz test random inputs
```

That's usually enough.

# A More Useful Standard

Instead of:

> "Can you explain every line?"

I think a better question is:

> "If this breaks in production, do we have enough understanding and evidence to diagnose and fix it?"

That's a much more practical bar.

For some code, that means deep implementation knowledge.

For other code, especially highly-tested and well-isolated code, it may only require understanding:

* intent
* behavior
* constraints
* failure modes

and not every implementation detail.

In fact, if an organization insists that every engineer must understand every implementation detail before approving a change, that organization will eventually struggle to scale. Modern software systems are simply too large. The more scalable approach is to require sufficient understanding **relative to the risk of the component**, while relying on tests, tooling, observability, and automation to provide the rest of the confidence.


--------------------------------------------------
========================================================

The middle ground is:

> **Let AI propose implementation, but humans own boundaries, contracts, risks, and acceptance evidence.**

Not:

> “AI may only write tiny functions.”

And not:

> “AI can build the whole thing and we’ll hope tests catch it.”

The right model is **contract-driven AI development**.

# 1. Separate Decisions From Implementation

AI can help with decisions, but should not silently own them.

## Humans should own

* Architecture boundaries
* Data ownership
* Security assumptions
* Error-handling policy
* Public APIs
* Persistence model
* Concurrency model
* Operational behavior
* “What happens when this fails?”

## AI can help with

* Implementation drafts
* Refactoring
* Test generation
* Edge-case discovery
* Documentation
* Comparing design options
* Finding bugs
* Producing examples

So the policy is not:

> “AI cannot make decisions.”

It is:

> “AI can recommend decisions, but humans must explicitly accept the important ones.”

---

# 2. Use an “AI Contract” Before Generating Code

Before asking AI to implement something, define the contract.

For Rust, this works very well:

```rust
/// Parses a user-supplied filter expression into a validated FilterAst.
///
/// Requirements:
/// - Accepts only documented operators: `and`, `or`, `not`, `=`, `!=`, `<`, `>`.
/// - Rejects unknown fields.
/// - Rejects malformed expressions with structured errors.
/// - Does not panic on invalid input.
/// - Does not allocate unbounded memory for hostile input.
/// - Does not perform I/O.
/// - Must be deterministic.
///
/// Failure behavior:
/// - Returns `Err(FilterParseError)` for invalid input.
/// - Never silently ignores invalid tokens.
pub fn parse_filter(input: &str) -> Result<FilterAst, FilterParseError> {
    todo!()
}
```

Now AI is not free-floating. It is filling in a constrained box.

That box is what the engineer reviews first.

---

# 3. The Engineer Does Not Need to Know Every Detail

The engineer needs to know enough to answer:

> “What is this supposed to do, what can go wrong, and how do we know it behaves correctly?”

That is different from:

> “Can you explain every internal line?”

For a complicated regex, parser, generated mapping table, SQL query, or serialization implementation, the bar should be:

* I understand the **intent**
* I understand the **inputs and outputs**
* I understand the **failure modes**
* I understand the **risk level**
* I have evidence from **tests / tooling / review**
* I know how to replace or isolate it if needed

That is sufficient in many cases.

---

# 4. Use a Risk-Based Understanding Standard

Not all code needs the same review depth.

| Code Type               |     AI Use | Human Understanding Required |
| ----------------------- | ---------: | ---------------------------- |
| Formatting/helper code  |       High | Low-Medium                   |
| DTOs/serde structs      |       High | Low-Medium                   |
| UI boilerplate          |       High | Medium                       |
| Business rules          |     Medium | High                         |
| Data migrations         |     Medium | High                         |
| Auth/security           | Low-Medium | Very High                    |
| Unsafe Rust             |        Low | Very High                    |
| Concurrency primitives  |        Low | Very High                    |
| Financial/payment logic | Low-Medium | Very High                    |
| Parsing untrusted input |     Medium | High                         |

The distinction is not “AI or not AI.”

The distinction is:

> **How expensive is being wrong?**

---

# 5. Define “Good Enough to Use”

Something is good enough to use when you have enough **evidence** relative to its risk.

For low-risk code:

```text
Requirements + normal unit tests + review
```

For medium-risk code:

```text
Requirements + docs + unit tests + edge cases + integration tests + reviewer explanation
```

For high-risk code:

```text
Requirements + design note + threat model + unit tests + property tests + fuzzing + observability + rollback plan
```

For critical code:

```text
All of the above + domain expert review + maybe avoid AI-generated implementation entirely
```

---

# 6. Treat AI Code Like a New Internal Dependency

This is the best analogy.

Serde is an external dependency with:

* reputation
* maintainers
* years of usage
* docs
* bug history
* ecosystem pressure

AI-generated code has none of that.

So you compensate with:

* smaller scope
* better tests
* clearer docs
* isolation
* explicit contracts
* reviewable design

You do **not** need to understand AI-generated code more because it is AI.

You need more evidence because it has no track record.

---

# 7. Make AI Code Isolated and Replaceable

A good AI-assisted implementation should have a narrow boundary.

Bad:

```text
AI changes architecture, data model, UI, validation, persistence, and error handling in one PR.
```

Good:

```text
AI implements `parse_filter()` behind a stable interface with tests.
```

The more isolated the code is, the less you need to understand every internal detail.

This is the same reason libraries work.

You trust the interface more than the internals.

---

# 8. Require an Explanation Artifact, Not Line-by-Line Understanding

For AI-assisted PRs, I would require a short section like:

```text
AI Assistance:
- Used to draft initial parser implementation.
- Human-defined API, error model, and requirements.
- Human-reviewed generated code for panic behavior, invalid input handling, and allocation behavior.

Risk:
- Parses untrusted input.
- Main risks are malformed input, excessive allocation, and incorrect operator precedence.

Evidence:
- Unit tests for valid/invalid expressions.
- Property tests for no-panic behavior.
- Fuzz target added for parser input.
- Integration test added for API behavior.
```

That is far more useful than asking:

> “Can you explain every line?”

---

# 9. Use AI in Layers

A good workflow:

## Step 1 — Human defines the problem

```text
We need a parser for X with these operators and these failure modes.
```

## Step 2 — AI proposes design options

```text
Give me 3 implementation approaches with tradeoffs.
```

## Step 3 — Human selects approach

```text
Use recursive descent. No parser generator. No unsafe. No panics.
```

## Step 4 — AI writes constrained implementation

```text
Implement only this module. Do not change public API.
```

## Step 5 — AI writes tests

```text
Generate unit tests, edge cases, property tests, and fuzz cases.
```

## Step 6 — Human reviews contract, risks, and evidence

Not every token. The system behavior.

---

# 10. The Final Standard

I would phrase the policy like this:

> Engineers are not required to understand every implementation detail of AI-assisted code. They are required to understand the code’s purpose, interface, assumptions, risks, failure modes, and evidence of correctness. The required depth of understanding scales with the risk and ownership of the code.

That is the middle ground.

It avoids both extremes:

* **No AI decision-making at all** → too conservative.
* **Vibe-coded black boxes** → reckless.

The mature standard is:

> **Human-owned contracts. AI-assisted implementation. Evidence-based approval. Risk-scaled review.**

## Existing Code Modification Rules

When modifying an existing codebase:

* Preserve existing style unless it is clearly broken.
* Minimize unrelated refactors.
* Do not change public APIs unless needed or requested.
* Do not rename things just for preference.
* Do not move large pieces of code unless it materially improves the design.
* Avoid broad rewrites when a focused change is enough.
* Keep behavior backwards-compatible unless the request requires a behavior change.
* Explain why each meaningful change exists.
* Prefer small, reviewable changes.
* Keep feature work, cleanup, and refactoring separate when possible.

If the codebase has a pattern, follow it. If the pattern is bad, explain the tradeoff before changing it.

---

## Architecture Preferences

Prefer clear, boring architecture.

Use:

* Small focused modules
* Small focused functions
* Explicit data types
* Config structs for configurable behavior
* Typed state instead of loose booleans when state has meaning
* Enums for meaningful modes, commands, events, or actions
* Structs for grouped data instead of long argument lists
* Dependency injection through constructors or function parameters
* Clear boundaries between domain logic, IO, UI, config, and external services

Avoid:

* Giant god functions
* Giant god structs
* Hidden global state
* Excessive mutation spread across unrelated modules
* Deeply nested control flow
* Stringly typed behavior when enums would be clearer
* Mixing UI code, business logic, IO, and parsing in the same function
* Overly abstract trait layers for small codebases

For larger codebases, prefer a hexagonal-style boundary:

* Domain/core logic should be mostly pure and easy to test.
* External systems should sit behind clear interfaces.
* UI should call into application/domain services instead of owning business rules.
* IO, network, filesystem, database, and GUI framework details should stay near the edges.

Do not force hexagonal architecture onto small/simple code. Use it when it improves maintainability.

---

## Error Handling Rules

Use idiomatic Rust error handling.

General rules:

* Return `Result<T, E>` when failure is expected or meaningful.
* Do not use `unwrap()` in production code unless the invariant is truly impossible and documented.
* Prefer `expect("specific reason")` over `unwrap()` when panicking is acceptable.
* Use `Option<T>` only when absence is a normal, non-error case.
* Use `Result<T, E>` when the caller needs to know why something failed.

Application-level errors:

* Use `anyhow::Result<T>` for errors that are not expected to be handled specifically by the caller.
* Add `.context("...")` or `.with_context(|| "...")` when the source of failure would otherwise be unclear.
* Context messages should explain what operation failed and include useful identifying information when available.

Example:

```rust
let contents = std::fs::read_to_string(path)
    .with_context(|| format!("failed to read config file at {}", path.display()))?;
```

Domain/library-level errors:

* Use explicit error types when the caller is expected to match on or recover from specific failures.
* Prefer a small custom error enum when the failure modes are meaningful.
* Use `thiserror` if it is already available or appropriate for the project.
* Keep error variants specific enough to be useful, but not so granular that they become noise.

Do not erase meaningful errors too early with `anyhow` if the caller needs to handle them.

---

## Documentation Style

I prefer detailed function documentation.

Use Rust doc comments for public functions, public structs, public enums, public traits, and important private functions.

For functions, prefer this structure when applicable:

```rust
/// Short summary of what the function does.
///
/// Longer explanation of why this function exists, what behavior it guarantees,
/// and any important implementation notes.
///
/// # Arguments
///
/// * `arg_name` - Explain what this input represents and any constraints.
///
/// # Returns
///
/// Explain what is returned on success. Be specific.
///
/// # Errors
///
/// Explain each meaningful error case. For `anyhow::Result`, explain the
/// operation that can fail and what context is attached.
///
/// # Panics
///
/// Include only if this function can panic.
///
/// # Examples
///
/// Include examples when useful, especially for reusable public APIs.
```

Document:

* Inputs
* Outputs
* Error behavior
* Edge cases
* Important invariants
* Caller responsibilities
* State changes
* Side effects
* Threading or async behavior, when relevant

Do not write useless documentation like “Gets the value” unless that is genuinely all the function does.

Good documentation should explain intent, constraints, and usage.

---

## Comment Style

Use moderate comments.

Comments should explain:

* Why something is done
* Non-obvious logic
* Edge cases
* State transitions
* Parsing decisions
* Error handling decisions
* GUI behavior that is not obvious from the code
* Async/concurrency behavior
* Safety invariants
* Workarounds for platform/framework behavior

Avoid comments that merely repeat the code.

Bad:

```rust
// Increment x by 1.
x += 1;
```

Good:

```rust
// Keep the retry count one-based because the value is shown directly in the UI.
attempt += 1;
```

Use heavier comments around:

* Parsing
* State machines
* Input routing
* Keyboard/mouse handling
* Async tasks
* FFI
* Unsafe code
* GUI event loops
* Cross-thread communication
* Error recovery paths

---

## Rust Style Preferences

Prefer:

* Clear names over short names
* `match` when handling meaningful branches
* `if let` / `let else` for simple early exits
* Early returns to reduce nesting
* Iterator chains when they remain readable
* Plain loops when iterator chains become too clever
* `derive(Debug, Clone, PartialEq, Eq)` where useful
* `Default` for reasonable default config/state
* `From` / `TryFrom` for meaningful conversions
* Newtypes when raw strings/integers would be ambiguous
* `Path` / `PathBuf` for filesystem paths
* `Duration` instead of raw millisecond integers inside runtime logic
* `Instant` for elapsed time
* `tracing` or existing logging patterns for diagnostics

Avoid:

* Excessive cloning
* Premature lifetimes that make code harder to use
* `Arc<Mutex<T>>` unless shared mutable state is truly needed
* Boolean parameter soup
* Magic numbers
* Overly generic functions without a concrete need
* Macros unless they materially reduce repetition
* Unsafe code unless there is no reasonable safe alternative

When using `clone()`, it should either be cheap, intentional, or explained.

---

## Async Rules

Async is allowed when appropriate.

Use async for:

* Network IO
* File IO when the project already uses async file APIs
* Timers
* Long-running tasks
* External service calls
* GUI background work, when supported by the app architecture

Do not make pure CPU/local logic async just because async exists.

Avoid holding locks across `.await`.

Clearly document task lifetime, cancellation behavior, and error handling for async code.

---

## GUI Code Guidelines

For GUI code, keep UI rendering and business logic separate where practical.

Prefer:

* UI code that reads state and emits commands/events
* Application logic that handles commands/events
* Clear state structs
* Clear config structs
* Small helper functions for repeated widgets
* Tooltips for non-obvious controls
* Explicit labels and descriptions
* Predictable state transitions
* Avoiding hidden side effects during rendering

Avoid:

* Large UI functions that also perform business logic
* Network or filesystem IO directly inside render/update functions
* Complex mutation scattered across widget code
* UI-only booleans that duplicate domain state
* Controls whose active/disabled state is not visually or logically clear

For `egui`/`eframe` style apps:

* Keep rendering functions focused.
* Use helper functions for panels, sections, tables, and controls.
* Keep long-running work out of the immediate UI update path.
* Prefer explicit commands/events when UI interactions affect deeper app state.

---

## Testing Expectations

Prefer smaller unit tests over full end-to-end tests unless the task specifically requires integration coverage.

Add or update tests for:

* Pure logic
* Parsing
* Formatting
* Config loading/defaulting
* Error cases
* Edge cases
* State transitions
* Regression cases for fixed bugs
* Table-driven behavior
* Boundary conditions

Use table-driven tests when multiple cases exercise the same logic.

Example:

```rust
#[test]
fn parses_known_actions() {
    let cases = [
        ("move_up", Action::MoveUp),
        ("move_down", Action::MoveDown),
    ];

    for (input, expected) in cases {
        let actual = Action::parse(input).expect("action should parse");
        assert_eq!(actual, expected);
    }
}
```

Use property tests when transformation/parsing logic has many possible cases and the project already supports property testing or it is clearly worth adding.

Do not add brittle tests that depend on incidental formatting unless formatting is the behavior being tested.

Do not fake tests. Do not claim tests were run unless they were actually run.

When suggesting tests, include the exact command that should be run, such as:

```text
cargo test
cargo test action_parser
cargo clippy --all-targets --all-features
cargo fmt
```

## Configuration Rules

When adding configuration:

* Add a clear config field name.
* Add a sensible default.
* Document what the field does.
* Document units, such as milliseconds, pixels, rows, seconds, or percentages.
* Keep parsed runtime types idiomatic, such as `Duration`, even if config uses integers.
* Validate invalid values early.
* Preserve backwards compatibility where possible.
* Update example config files and comments.

Avoid hardcoded runtime behavior when the code already has a config system for that category.

---

## Logging / Observability

Use logging when it helps diagnose behavior.

Prefer existing project logging patterns. If the project uses `tracing`, use `tracing`.

Good places for logs:

* Config loading failures
* Fallback behavior
* External service failures
* Retry behavior
* Input routing decisions
* State transitions
* Background task failures
* Recoverable errors

Do not spam logs in hot loops unless throttled or debug-level.

---

## Safety and Invariants

Avoid `unsafe`.

If unsafe is required:

* Keep it as small as possible.
* Encapsulate it behind a safe API.
* Document the safety requirements.
* Explain why safe Rust is insufficient.
* Add tests around the safe wrapper when practical.

For invariants, make invalid states unrepresentable where reasonable.

Prefer:

* Enums over loose strings
* Newtypes over ambiguous primitive values
* `NonZero*` types when zero is invalid
* Validated constructors when values have constraints

---

## Performance Guidance

Do not prematurely optimize.

Prefer readable code first.

Optimize when:

* The code is in a hot path
* The data size is large
* The user explicitly asks for performance
* The existing code shows performance sensitivity
* The algorithmic complexity is clearly bad

When optimizing, explain the tradeoff.

Avoid unnecessary allocation in hot paths, but do not make ordinary code harder to read just to avoid tiny allocations.

Common crates are acceptable when present or justified, including:

* `anyhow`
* `thiserror`
* `serde`
* `serde_json`
* `toml`
* `tracing`
* `tokio`
* `clap`
* `uuid`
* `derive_more`

Do not assume a crate is available without checking.

Do not add a dependency for something the standard library handles cleanly.

The plan should include:

* What will change
* Which files/modules are likely involved
* Data structures to add or modify
* Error handling strategy
* Testing strategy
* Risks or compatibility concerns

Then provide the code.

For very small tasks, a short explanation plus code is enough.

## 2. Implementation Plan Contract

Use this when you want **tasks, not code yet**. This matches how you often ask for “implementation steps with detail.”

# Implementation Plan Contract

I want an implementation plan, not code yet.

Create a detailed, step-by-step plan for the requested change. The plan should be specific enough that another developer or coding agent could implement it safely.

## Required Structure

For each task, include:

* **Goal** - What this task accomplishes
* **Why** - Why the change is needed
* **Files likely involved** - Specific files or modules if known
* **What to change** - Concrete implementation details
* **Data structures / fields** - Any structs, enums, config fields, or function signatures likely needed
* **Behavioral rules** - Exact runtime behavior expected
* **Edge cases** - Important cases to handle
* **Tests** - Unit/regression/table-driven tests to add
* **Validation** - Commands or manual checks to run
* **Risk level** - Low, medium, or high, with explanation

## Planning Rules

Prefer granular tasks over large vague tasks.

Avoid saying “refactor the system” without explaining exactly what should move, where it should move, and why.

Separate these categories when possible:

1. Model/data changes
2. Config changes
3. Runtime behavior changes
4. UI changes
5. Error handling changes
6. Tests
7. Documentation
8. Cleanup

## Style

Be direct, precise, and implementation-oriented.

Do not provide code unless I explicitly ask for code after the plan.

If there are multiple valid approaches, compare them and recommend one.

## 3. Debugging / Root-Cause Contract

Use this when something is broken and you want the model to avoid random guessing.

# Debugging and Root-Cause Analysis Contract

I want root-cause analysis, not random guesses.

Analyze the bug systematically.

## Required Response Structure

1. **Observed behavior**

   * Restate the bug in precise terms.

2. **Expected behavior**

   * State what should happen instead.

3. **Most likely root causes**

   * List likely causes in priority order.
   * Explain why each cause fits the symptoms.

4. **Evidence to inspect**

   * Name the files, functions, logs, state transitions, config fields, or tests that should be checked.

5. **Root cause**

   * Identify the most likely actual cause.
   * Be explicit if the evidence is incomplete.

6. **Fix plan**

   * Give concrete implementation steps.
   * Prefer minimal, targeted fixes.

7. **Regression tests**

   * Explain what tests should be added so this bug does not come back.

8. **Validation**

   * Provide commands and manual checks.

## Debugging Rules

Do not jump straight to code.

Do not rewrite unrelated systems unless the architecture is directly causing the bug.

Preserve working behavior.

When comparing a “good” and “bad” version of code:

* Identify behavioral differences first.
* Look for changed timing, input handling, state ownership, event loops, threading, debounce/throttle behavior, and config defaults.
* Explain why the good version works.
* Explain why the bad version fails.
* Propose the smallest fix that restores the working behavior while preserving desired new features.

## Output Preference

When code changes are eventually needed, group them by file path and include enough context to place the code.

# Code Explanation Contract

I want you to explain this Rust code clearly and accurately.

Assume I am an experienced programmer, but I may not know this specific codebase yet.

## Required Explanation Structure

Explain the code in this order:

1. **High-level purpose**

   * What problem this code solves.
   * Where it likely fits in the project.

2. **Main data types**

   * Explain important structs, enums, traits, type aliases, and constants.
   * Explain what each type represents.
   * Explain any invariants or assumptions.

3. **Control flow**

   * Walk through the important functions in execution order.
   * Explain how data moves through the system.
   * Explain important branches and early returns.

4. **Error handling**

   * Explain what can fail.
   * Explain whether failures are handled locally or returned to the caller.
   * Explain whether `anyhow`, explicit errors, `Option`, `Result`, `unwrap`, or `expect` are used appropriately.

5. **State changes**

   * Explain what mutable state exists.
   * Explain who owns it.
   * Explain when and why it changes.

6. **Edge cases**

   * Identify boundary cases, invalid inputs, empty inputs, missing config, timing issues, or state conflicts.

7. **Tests**

   * Explain what behavior is currently tested.
   * Identify missing tests.

8. **Potential improvements**

   * List concrete improvements, but separate them from the explanation.

## Explanation Rules

Do not simply restate the code line by line.

Focus on intent, behavior, ownership, state, errors, and design.

Be explicit when something is unclear or inferred.

Do not invent behavior that is not supported by the code.

# Devil's Advocate / Design Challenge Contract

I want you to challenge this Rust design or implementation.

Act as a skeptical senior engineer reviewing the approach before it becomes permanent.

## Required Structure

1. **Restate the design**

   * Briefly describe what the design appears to be doing.

2. **What is good about it**

   * Identify the strengths first.

3. **Main concerns**

   * List the strongest objections.
   * Focus on maintainability, correctness, testability, performance, state flow, API design, and error handling.

4. **Failure modes**

   * Explain how this design could break in production or become painful later.

5. **Over-engineering check**

   * Identify anything that seems more complex than the problem requires.

6. **Under-engineering check**

   * Identify anything that is too ad hoc, too implicit, or likely to become technical debt.

7. **Rust-specific concerns**

   * Discuss ownership, lifetimes, cloning, trait boundaries, error types, async behavior, lock usage, and type safety.

8. **Alternative designs**

   * Provide 2-3 alternatives.
   * Explain tradeoffs.
   * Recommend one.

9. **Decision**

   * State whether you would approve, revise, or reject the design.
   * Give concrete changes required for approval.

## Rules

Do not be contrarian for no reason.

Do not recommend a complex architecture unless the complexity is justified.

Prefer concrete changes over abstract design advice.

Be honest about tradeoffs.

# Next Steps / Feature Roadmap Contract

I want practical next steps for this Rust project or feature.

Do not give vague suggestions. Give implementation-ready tasks.

## Required Structure

Group suggestions into these categories:

1. **Correctness**

   * Bugs, edge cases, validation, state conflicts, regression risks.

2. **Maintainability**

   * Module boundaries, duplicated logic, confusing functions, naming, type design.

3. **Testing**

   * Unit tests, table-driven tests, regression tests, property tests if useful.

4. **User experience**

   * UI, tooltips, config clarity, error messages, logs, diagnostics.

5. **Performance**

   * Hot paths, unnecessary allocations, excessive IO, expensive recomputation.

6. **Observability**

   * Logging, tracing, debug output, metrics, diagnostics.

7. **Documentation**

   * README, config comments, examples, function docs, architecture notes.

8. **Future features**

   * Useful features that build naturally on the current design.

## For Each Suggested Task

Include:

* **Task name**
* **Why it matters**
* **Files likely involved**
* **What to change**
* **Tests to add**
* **Risk level**
* **Estimated complexity**
* **Whether it should be done now or later**

## Rules

Prioritize high-value, low-risk improvements first.

Separate must-fix issues from nice-to-have ideas.

Do not suggest broad rewrites unless there is a clear payoff.

# Refactor Safety Contract

I want a safe refactor plan, not a rewrite.

The goal is to improve structure while preserving behavior.

## Required Structure

1. **Current pain points**

   * Identify concrete maintainability problems.

2. **Behavior that must not change**

   * List existing behavior that should remain identical.

3. **Refactor boundaries**

   * Identify what should move and what should stay.

4. **Step-by-step refactor plan**

   * Each step should be independently reviewable.
   * Each step should keep the project compiling.

5. **Tests before refactor**

   * Identify tests that should exist before changing structure.

6. **Tests after refactor**

   * Identify tests that prove behavior was preserved.

7. **Risk areas**

   * Identify where regressions are most likely.

8. **Rollback strategy**

   * Explain how to keep changes easy to revert.

## Rules

Do not combine refactoring with feature changes unless explicitly requested.

Do not rename or move things just for aesthetics.

Prefer extracting pure logic first.

Prefer adding tests before changing behavior-heavy code.

# API / Type Design Contract

I want help designing Rust types and APIs.

Focus on making invalid states hard to represent and caller intent clear.

## Required Structure

1. **Domain concepts**

   * Identify the real concepts that need types.

2. **Proposed types**

   * Suggest structs, enums, newtypes, traits, and config types.

3. **Ownership model**

   * Explain who owns each value.
   * Explain where borrowing is appropriate.
   * Explain where cloning is acceptable.

4. **Function boundaries**

   * Propose public functions and important private helpers.
   * Explain inputs, outputs, and errors.

5. **Error model**

   * Identify which errors should be explicit typed errors.
   * Identify where `anyhow::Result` is appropriate.

6. **State model**

   * Explain mutable state and transitions.
   * Prefer enums for modes/states instead of loose booleans.

7. **Extensibility**

   * Explain how this design can grow without major rewrites.

8. **Tests**

   * Identify unit tests for each type or function.

## Rules

Prefer clear types over clever generic abstractions.

Do not introduce traits unless there are multiple implementations, a testing need, or a real boundary.

Use enums for meaningful modes, commands, actions, and states.

Use newtypes when raw strings, numbers, or IDs would be ambiguous.

# State Machine Review Contract

I want a state-machine-oriented review of this Rust code.

This is especially important for input handling, GUI behavior, keyboard/mouse control, timers, modes, and command routing.

## Required Structure

1. **States**

   * Identify all meaningful states.
   * Identify whether they are explicit or implicit.

2. **Events**

   * Identify inputs/events that cause state changes.

3. **Transitions**

   * Explain valid transitions.
   * Identify invalid or missing transitions.

4. **Priority rules**

   * Explain which events/actions should win when multiple things happen.

5. **Stuck states**

   * Identify ways the app can get trapped in a bad state.

6. **Race/timing risks**

   * Identify debounce, repeat, timeout, async, or ordering problems.

7. **Recommended model**

   * Suggest enums, structs, or routing changes to make state explicit.

8. **Tests**

   * Recommend tests for transitions, priority, cancellation, and recovery.

## Rules

Prefer explicit state over scattered booleans.

Prefer central routing for priority-sensitive input.

Emergency/exit/kill-switch behavior should be highest priority if applicable.

# Async / Concurrency Review Contract

I want a Rust async/concurrency review.

Focus on correctness, cancellation, ownership, task lifetime, channels, locks, and GUI responsiveness.

## Required Structure

1. **Async boundaries**

   * Identify what is async and why.

2. **Task ownership**

   * Explain who spawns tasks and who owns their handles.

3. **Cancellation**

   * Explain how tasks stop.
   * Identify tasks that can leak or outlive their owner.

4. **Shared state**

   * Identify `Arc`, `Mutex`, `RwLock`, channels, atomics, or shared state.
   * Check whether locks are held across `.await`.

5. **Error propagation**

   * Explain where task errors go.
   * Identify swallowed errors.

6. **Backpressure**

   * Identify unbounded queues, event floods, or hot loops.

7. **GUI responsiveness**

   * Identify blocking work that could freeze the UI.

8. **Recommendations**

   * Provide concrete changes and tests.

## Rules

Do not make pure logic async unnecessarily.

Avoid holding locks across `.await`.

Prefer clear task lifetime and cancellation behavior.

# Acceptance Criteria Contract

Before implementing, define what “done” means.

## Required Structure

1. **Feature summary**
2. **Functional requirements**
3. **Non-functional requirements**
4. **Out-of-scope items**
5. **Edge cases**
6. **Error behavior**
7. **Config behavior**
8. **UI behavior**, if applicable
9. **Tests required**
10. **Manual validation steps**
11. **Definition of done**

## Rules

Acceptance criteria should be testable.

Avoid vague requirements like “make it better.”

Use concrete statements like:

* “When X happens, Y should occur.”
* “If config value Z is missing, default to A.”
* “If parsing fails, return error B with context C.”
* “The UI should show disabled state when D is unavailable.”
